What breaking AWS taught me about breaking AI

2026-07-05

I spend half my time breaking into AWS environments and the other half poking at AI systems — the LLM apps and agents companies are shipping as fast as they can write the prompt. On paper those are different jobs. In my head they're the same one. The instinct that finds a privilege-escalation path through an IAM policy is the exact instinct that finds a prompt injection. Both come down to one question: where does this system's authority outrun its intent?

Let me show you what I mean, because once you see it you can't unsee it.

The confused deputy never went away

Cloud pentesters have a favorite bug class: the confused deputy. You find some component — a Lambda, a CI role, an SSRF-able metadata endpoint — that holds more permission than the caller should have, and you trick it into acting on your behalf. You're not authorized. The deputy is. So you make the deputy do it.

Prompt injection is a confused deputy attack. That's the whole trick. An LLM agent with tools is a deputy: it holds real authority — it can read a database, send an email, call an internal API — and it takes instructions from text. When some of that text comes from an attacker (a web page it summarizes, an email it reads, a support ticket it triages), the attacker is now writing instructions for a deputy that outranks them. Same bug, new substrate. We even solved it before, in a narrow way, with SSRF and IMDSv2. We just forgot the lesson before the next deputy showed up.

Wildcards, but in English

The first thing I look for in an IAM policy is "Action": "*" or a resource ARN that's broader than the job in front of it. Over-broad grants are where cloud compromises live — not because someone meant to hand out admin, but because scoping is tedious and * is easy.

AI tool definitions are IAM policies written in English, and they're full of wildcards nobody labels as wildcards. "The agent can run shell commands." "The agent has read access to the wiki." "It can query the database to answer questions." Every one of those is an Action: * waiting for an attacker-supplied instruction to aim it. The blast radius of a prompt injection isn't set by the model — it's set by the scope of the tools you handed the deputy. Give an agent a wildcard and injection becomes privilege escalation.

The lesson transfers cleanly

Here's the part that actually matters, and it's why I'm not scared of AI agents so much as I'm scared of how we're wiring them up. The mitigations are the ones we already know:

Least privilege at the boundary. Don't ask "can the model be tricked?" — assume it can, the same way you assume any input-taking component can be confused. Ask instead: when it is tricked, what can the deputy actually do? Scope the tools like you'd scope a role. Read-only where you can. Narrow resources. No standing write access to anything you'd cry about.

Don't let the caller's input define authority. In cloud, you never let the request decide its own permissions. Same rule: instructions and data need to live in different trust tiers, and the tier that came from outside must never be the tier that grants power. Most shipped agents collapse those two into one context window and call it a feature.

Put the dangerous step behind a gate a human owns. The stuff I can't undo — a wire transfer, a delete, an outbound email — sits behind an explicit confirm, not because the model is dumb but because deputies get confused and the cost of that confusion should be bounded. (I build my own assistant this way: it can read and draft all day, but anything that leaves the house or can't be taken back waits for me to say go.)

Same job, wider blast radius

None of this is a new discipline. It's threat modeling, applied to a component that happens to speak English and improvise. The people who'll be good at securing AI systems aren't the ones who know the most about transformers — they're the ones who already have the reflex to ask where authority outruns intent, and who've watched that gap get exploited a hundred times in a hundred other systems.

The models are new. The mistake isn't. We're handing deputies broad authority and trusting untrusted input to guide them, and we're calling it progress. It's the oldest bug we have, wearing a very convincing new coat.

I'll be writing more of these — cloud, AI, the seam between them. If that's your seam too, stick around.
